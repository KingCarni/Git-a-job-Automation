## Goal

Build a first-party resume parser foundation so Git-a-Job can reliably accept PDF/DOCX resumes, produce structured resume data, and avoid broken previews when parsing confidence is low.

## Why this matters

Resume upload is a critical entry point. If PDF parsing fails or produces mangled output, users may abandon the product before reaching the resume compiler, job match, or cover letter flows.

Current problem examples:

- PDF text can be extracted, but downstream parsing fails to structure the resume.
- Normal resume date formats such as `May 2017 to Present`, `November 2013 to June 2016`, and `01/2015-06/2017` are not handled reliably.
- Bullet glyphs such as `‚Ä¢`, `ÔÇß`, `‚ñ™`, `‚ó¶`, `*`, and hyphen bullets are inconsistently detected.
- Wrapped bullet continuation lines are not safely rebuilt.
- Section aliases such as `Employment History` are not consistently treated as experience.
- The UI can show a broken/mangled preview instead of a useful review fallback.

## Strategic direction

Do not keep patching resume parsing as a helper buried inside `/api/analyze`.
Create a dedicated internal parser subsystem inspired by mature parser models such as Textkernel Tx output shapes, but implemented in our own TypeScript code.

Textkernel SDK audit takeaway:

- The SDK exposes access to a paid API parser, not an open-source local parser engine.
- Useful design ideas to borrow: structured ParsedResume model, EmploymentHistory/Position/Bullet objects, parser metadata, confidence/probability fields, and partial-success handling.

## Phase plan / subtasks

### 1. Parser foundation and types

Create parser types and an orchestration entry point. Return both structured data and diagnostics. Do not replace the whole analyzer yet; integrate behind a compatibility layer.

Acceptance criteria:

- `parseResumeDocument(rawText, metadata)` returns a stable `ParsedResumeDocument`.
- Parser returns warnings and confidence instead of throwing on partial parse failures.
- Existing analyze flow can still consume a flattened compatibility output.

### 2. Text normalization

Normalize messy PDF/DOCX extracted text before parsing.

Must handle:

- bullet glyphs: `‚Ä¢`, `‚óè`, `ÔÇß`, `‚ñ™`, `‚ó¶`, `‚Ä£`, `*`, `-`
- dash variants: `‚Äì`, `‚Äî`, `-`
- date connectors: `to`, `through`, `-`, `‚Äì`
- repeated whitespace
- broken line wrapping
- duplicate blank lines

Acceptance criteria:

- Bullet-like lines are normalized to a consistent internal marker.
- Existing fake resume PDFs retain useful line structure after normalization.
- Normalization does not destroy contact lines, dates, or section headings.

### 3. Section detection

Add deterministic section detection with aliases.

Must detect:

- Profile / Summary / Professional Summary / Objective
- Experience / Work Experience / Professional Experience / Employment History / Work History / Career History / Relevant Experience
- Education
- Skills / Technical Skills / Core Skills / Areas of Expertise
- Certifications / Licenses
- Projects
- Interests / Activities

Acceptance criteria:

- `Employment History` is treated as experience.
- Missing sections do not crash the parser.
- Parser records which section headings were detected.

### 4. Experience block parser

Parse experience into position blocks instead of relying only on global bullet detection.

Must support date formats:

- `May 2017 to Present`
- `May 2017 - Present`
- `November 2013 to June 2016`
- `01/2015-06/2017`
- `01/2015 - 06/2017`
- `2015 - 2017`
- `2017 ‚Äì Present`

Must support common header layouts:

- Title line followed by company/date line
- Date first, then title/company
- Company - Location - Date range
- Title ‚Äì Company ‚Äì Date range

Acceptance criteria:

- The Homer fake resume parses `Night Auditor` as a position with `Springfield Inn` and bullets.
- The simple fake resume parses `Host` and `Food Service Management Intern` as positions.
- Wrapped bullet continuation lines attach to the previous bullet where appropriate.

### 5. Contact, education, and skills extraction

Implement basic deterministic extraction for contact, education, and skills.

Acceptance criteria:

- Email and phone are extracted when present.
- Education section is preserved at least as structured text entries.
- Skills list is extracted when a skills section exists.
- No sensitive resume text is logged.

### 6. Parse quality scoring and partial-success behavior

Score parse quality so the UI can decide whether to render a preview or ask the user to review extracted text.

Suggested quality metrics:

- textLength
- sectionCount
- positionCount
- bulletCount
- dateCount
- contactSignalsFound
- warningCount

Acceptance criteria:

- Parser returns `high`, `medium`, or `low` confidence.
- Low confidence does not hard-fail if text was extracted.
- Parser distinguishes true extraction failure from usable-but-needs-review parsing.

### 7. UI review fallback

When confidence is low, avoid showing a mangled preview.

Acceptance criteria:

- User sees a clear message such as: `Your resume text was extracted, but the structure needs review.`
- User can inspect/edit extracted text before continuing.
- No more scary `No bullets detected` dead-end for usable text.

### 8. Analyzer integration

Wire parser output into existing resume analysis/profile creation carefully.

Acceptance criteria:

- Existing resume compiler still works for good DOCX/text cases.
- PDF cases use the new parser diagnostics.
- ResumeProfile creation can consume structured parser output where available.
- Compatibility output remains available for older code paths.

### 9. Fixtures and regression coverage

Use the uploaded fake PDFs / extracted text as parser regression fixtures.

Acceptance criteria:

- Add tests or fixture assertions for both fake resumes.
- Tests verify section detection, position detection, bullet count, and date parsing.
- Future parser changes should not silently regress these cases.

## Out of scope for first pass

- Paid parser API integration
- Full OCR pipeline
- LLM-based resume repair
- Major UI redesign
- Rewriting resume templates
- Job matching changes

## Future phases

- PDF layout-aware extraction using text item coordinates
- OCR fallback for scanned/image PDFs
- Optional AI repair pass for low-confidence parses
- Skill provenance weighting for job matching
- User-facing profile correction UI

## QA notes

Primary manual QA should use:

- Fake Resume / Homer Simpson PDF
- Fake-Resume template PDF
- A clean DOCX resume
- A pasted plain-text resume

Success means the parser either produces a reasonable structured resume or gives a useful review fallback instead of broken preview output.
