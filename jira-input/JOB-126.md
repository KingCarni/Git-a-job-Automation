Implementation plan refinement for JOB-126



The idea is strong and worth doing. This should be treated as a lightweight application-tracking layer inside the jobs workspace, not a full ATS or CRM.



Recommended product shape



Keep one jobs workspace with two tabs/views:



Saved



Applied



Applied jobs should remain visible in feed/detail but be clearly marked as Applied. Do not auto-remove them from the feed by default. Add an optional hide/filter behavior later if needed.



Recommended status model



Saved



Applied



Interview



Offer



Rejected



Archived



Recommended analytics scope Track state transitions, but keep analytics as support scope rather than the center of the ticket. Track at minimum:



marked applied



moved to interview



moved to offer



marked rejected



archived



MVP recommendation



For this ticket, focus on the smallest real workflow:



User can mark a saved or visible job as Applied.



User can view Applied jobs under a separate Applied tab/view in the jobs workspace.



User can change an applied job status to Interview, Offer, Rejected, or Archived.



Feed and detail pages clearly show when a job is already Applied and what status it is in.



State transition analytics fire for the key transitions above.



What not to do yet



no kanban board



no freeform notes/history timeline unless it falls out almost free



no reminder system yet



no auto-removal from feed as default behavior



no giant multi-step application CRM



Suggested implementation split



Preferred as two tickets or two internal phases if we keep one Jira ticket:



Phase A — Application state model + actions



add DB model for per-user job application state



add mark-applied / update-status actions or routes



show applied/status badge on feed and detail



ensure applied jobs are distinct from saved jobs



Phase B — Jobs workspace tabs + analytics



make jobs workspace support Saved and Applied views/tabs



render applied list cleanly



wire key state transition analytics



keep analytics simple but structured enough for future feedback work



Suggested data model



Add a dedicated model instead of overloading SavedJob or HiddenJob.



Example direction: JobApplication



id



userId



jobId



status



createdAt



updatedAt



appliedAt nullable if needed



Constraint:



unique(userId, jobId)



Behavior notes



Saved and Applied should be conceptually separate.



A job can be saved before applied, but once application tracking exists, the applied state should be owned by the JobApplication record.



Rejected and Archived should not mean destructive delete.



Unknown or stale jobs should still render gracefully in the Applied view.



Suggested acceptance criteria



Jobs workspace has separate Saved and Applied tabs/views.



User can mark a job as Applied from feed/detail.



Applied jobs remain visible in feed/detail and are clearly labeled.



User can move an applied job between Applied, Interview, Offer, Rejected, and Archived.



Applied jobs are visible in the Applied view.



Key application state transitions emit analytics events.



This does not turn the jobs feature into a heavy CRM.



Likely implementation surfaces



Prisma schema for JobApplication



jobs feed/detail API queries



saved/applied workspace page or tabs



small status-update action/route



analytics event wiring



Blunt recommendation Build the clean MVP first: separate Saved and Applied views, explicit application states, visible Applied badges in feed/detail, and state transition analytics. That gets the real value without dragging the product into fake complexity.
